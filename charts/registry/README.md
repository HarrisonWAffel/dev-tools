# registry

A minimal private Docker registry deployed on Kubernetes using the official [`registry:2`](https://hub.docker.com/_/registry) image.

Features:
- Optional htpasswd authentication (passwords generated by Helm — no external tools needed)
- Optional pull-through cache mode for proxying a remote registry (e.g. Docker Hub)
- Persistent storage via the Rancher LocalPath Provisioner (built into RKE2)
- ClusterIP service with an annotations hook for future external access integration
- Shared-storage support for running a push registry and pull-through proxy side by side

## Prerequisites

- Helm 3
- Kubernetes cluster with RKE2 (or any cluster with the `local-path` StorageClass available)

## Installation

### Basic (no auth, no persistence to disk concerns)

```bash
helm install registry . -n registry --create-namespace
```

### With authentication

Specify one or more username/password pairs. Helm generates the bcrypt hashes automatically.

```bash
helm install registry . -n registry --create-namespace \
  --set auth.enabled=true \
  --set 'auth.users[0].username=myuser' \
  --set 'auth.users[0].password=mypassword'
```

Or via a values file (recommended for multiple users):

```yaml
# my-values.yaml
auth:
  enabled: true
  users:
    - username: alice
      password: s3cur3p@ss
    - username: bob
      password: an0th3rp@ss
```

```bash
helm install registry . -n registry --create-namespace -f my-values.yaml
```

### As a pull-through cache

When proxy mode is enabled, the registry transparently pulls images from a remote registry on demand. **Pushing images is not supported in proxy mode** (this is a Docker registry limitation).

```bash
helm install registry . -n registry --create-namespace \
  --set proxy.enabled=true \
  --set proxy.remoteUrl=https://registry-1.docker.io
```

Proxy mode and authentication can be enabled together.

### Push registry + pull-through proxy (shared storage)

A useful pattern for testing is to run two registry instances that share the same backing storage:

- **Push registry** — accepts `docker push`. Stores your private images.
- **Proxy registry** — pull-through cache pointed at Docker Hub (or another upstream). Also serves the private images stored by the push registry, because they share the same disk.

This gives you a single endpoint for `docker pull` that transparently serves both cached public images and your own private ones, without needing to pre-load anything.

**Step 1 — Install the push registry** (creates the PVC):

```bash
helm install push . -n registry --create-namespace \
  --set auth.enabled=true \
  --set 'auth.users[0].username=alice' \
  --set 'auth.users[0].password=s3cur3p@ss'
```

**Step 2 — Install the proxy registry**, pointing it at the push registry's PVC:

```bash
helm install proxy . -n registry \
  --set proxy.enabled=true \
  --set proxy.remoteUrl=https://registry-1.docker.io \
  --set persistence.existingClaim=push-data
```

> The PVC name for a release is `<release-name>-data`, so for the `push` release above it is `push-data`.

You now have two services in the `registry` namespace:
- `push.registry.svc.cluster.local:5000` — push private images here
- `proxy.registry.svc.cluster.local:5000` — pull from here (serves both Docker Hub and your private images)

**Pushing a private image:**

```bash
kubectl port-forward -n registry svc/push 5000:5000

docker login localhost:5000 -u alice -p s3cur3p@ss
docker tag myapp:latest localhost:5000/myapp:latest
docker push localhost:5000/myapp:latest
```

**Pulling via the proxy:**

```bash
kubectl port-forward -n registry svc/proxy 5000:5000

# Pull a public image (cached from Docker Hub on first pull)
docker pull localhost:5000/library/alpine:latest

# Pull your private image (served from shared storage)
docker pull localhost:5000/myapp:latest
```

> **Note:** The proxy registry does not enforce authentication, so it is suitable for read-only cluster-internal access. If you want to protect pull access as well, enable `auth` on the proxy install too.

### Using a pre-existing htpasswd Secret

If you prefer to manage credentials outside of Helm (e.g. via an external secrets operator), create a Kubernetes Secret with an `htpasswd` key and reference it:

```bash
# Create the secret manually
kubectl create secret generic my-registry-creds \
  --from-literal=htpasswd='alice:$2y$05$...' \
  -n registry

# Reference it in the chart
helm install registry . -n registry --create-namespace \
  --set auth.enabled=true \
  --set auth.existingSecret=my-registry-creds
```

## Accessing the registry

The registry is exposed as a ClusterIP service on port 5000. It is not reachable from outside the cluster by default.

**From within the cluster**, other pods can reach it at:

```
<release-name>.<namespace>.svc.cluster.local:5000
```

For example, with the default release name `registry` in the `registry` namespace:

```
registry.registry.svc.cluster.local:5000
```

**For local access during development**, use port-forward:

```bash
kubectl port-forward -n registry svc/registry 5000:5000
```

Then push/pull against `localhost:5000`.

**For external access**, set `service.annotations` to whatever your ingress or tunnel solution (e.g. Pangolin) expects, or change `service.type` to `NodePort` or `LoadBalancer`.

## Smoke test

After installing, verify the registry is healthy:

```bash
# Port-forward in one terminal
kubectl port-forward -n registry svc/registry 5000:5000

# In another terminal — no auth
curl http://localhost:5000/v2/
# expected: {}

# With auth enabled
curl -u alice:s3cur3p@ss http://localhost:5000/v2/
# expected: {}

# Push a test image
docker tag alpine localhost:5000/alpine:test
docker push localhost:5000/alpine:test

# Pull it back
docker pull localhost:5000/alpine:test
```

## Upgrading

```bash
helm upgrade registry . -n registry -f my-values.yaml
```

> **Note on auth upgrades**: Because Helm re-generates bcrypt hashes on each render, upgrading with `auth.users` will always update the Secret and trigger a pod restart. This is expected and safe. To avoid restarts on upgrade, switch to `auth.existingSecret` with a manually managed Secret.

## Uninstalling

```bash
helm uninstall registry -n registry
```

The PersistentVolumeClaim is **not** deleted automatically. To remove stored registry data:

```bash
kubectl delete pvc -n registry registry-data
```

## Values reference

| Key | Default | Description |
|-----|---------|-------------|
| `image.repository` | `registry` | Container image repository |
| `image.tag` | `"2"` | Image tag |
| `image.pullPolicy` | `IfNotPresent` | Image pull policy |
| `replicaCount` | `1` | Number of replicas (keep at 1 with `ReadWriteOnce` storage) |
| `service.type` | `ClusterIP` | Kubernetes service type |
| `service.port` | `5000` | Service port |
| `service.annotations` | `{}` | Annotations on the Service resource |
| `persistence.enabled` | `true` | Enable persistent storage |
| `persistence.storageClassName` | `local-path` | StorageClass name |
| `persistence.accessMode` | `ReadWriteOnce` | PVC access mode |
| `persistence.size` | `10Gi` | PVC storage size |
| `persistence.existingClaim` | `""` | Use an existing PVC instead of creating one (see shared-storage setup) |
| `auth.enabled` | `false` | Enable htpasswd authentication |
| `auth.users` | `[]` | List of `{username, password}` objects |
| `auth.existingSecret` | `""` | Name of a pre-existing Secret with an `htpasswd` key |
| `proxy.enabled` | `false` | Enable pull-through cache mode |
| `proxy.remoteUrl` | `https://registry-1.docker.io` | Upstream registry URL |
| `extraEnv` | `[]` | Extra environment variables for the registry container |
| `resources` | `{}` | CPU/memory requests and limits |
| `nodeSelector` | `{}` | Node selector constraints |
| `tolerations` | `[]` | Pod tolerations |
